# 1. 两数之和

> LeetCode 1 · 简单

## 题目

给定一个整数数组 `nums` 和一个整数目标值 `target`,在数组中找出 **和为 `target`** 的两个整数,返回它们的下标。

- 每种输入只对应一个答案
- 同一个元素不能使用两次

```
输入: nums = [2,7,11,15], target = 9   输出: [0,1]
输入: nums = [3,3],       target = 6   输出: [0,1]
```

---

## 我的代码

```cpp
class Solution {
public:
    vector<int> twoSum(vector<int>& nums, int target) {
        vector<int> index;
        index.reserve(2);                          // ← 错在这
        for (int i = 0; i < nums.size() - 1; i++) {
            index[0] = i;                          // ← 越界写
            int answer = target - nums[i];
            for (int j = i + 1; j < nums.size(); j++) {
                if (nums[j] == answer) {
                    index[1] = j;                  // ← 越界写
                    return index;
                }
            }
        }
        return index;
    }
};
```

---

## 我犯的错

### 错误 1:把 `reserve` 当成了 `resize`

我以为 `reserve(2)` 之后 vector 里就有 2 个位置可以用 `index[0] = ...` 赋值了。**错了。**

| 方法 | 实际作用 | 调用后 `v[0]` 能用吗 |
|---|---|---|
| `v.reserve(2)` | 只预留**容量**(底层内存),`size()` 仍是 0,vector 里**没有元素** | ❌ 未定义行为 |
| `v.resize(2)` | 真的把 `size()` 改成 2,创建 2 个默认值 | ✅ |
| `v.push_back(x)` | 末尾追加一个元素,`size()` +1 | ✅ |

`reserve` 只是告诉 vector"我打算装 2 个,你先备好内存,别反复扩容",**vector 里并没有元素**。`index[0] = i` 写入了一块"属于 vector 但没激活"的内存——这是**未定义行为**,判题机因此判错。

**记忆口诀**:`reserve` 是预定座位(没人坐),`resize` 是直接坐进去。

### 错误 2:`size() - 1` 的无符号下溢

```cpp
for (int i = 0; i < nums.size() - 1; i++)
```

`nums.size()` 返回的是 `size_t`(无符号整数)。如果哪天 `nums` 是空的,`0 - 1` **不会**得到 `-1`,而是会变成 `size_t` 的最大值(约 1.8×10¹⁹),循环直接炸。

本题题目保证 `nums.length >= 2`,这次没踩到,但下次不一定有保护。

**习惯**:无符号数做减法前,先确认被减数 ≥ 减数,或者改写成 `i + 1 < nums.size()`。

### 错误 3:用了笨办法(O(n²))

我写的是双层循环。能过,但不是这题的"标准答案",见下文哈希表解法。

## 修好的暴力版本(O(n²))

先把 bug 修了,跑通再说:

```cpp
class Solution {
public:
    vector<int> twoSum(vector<int>& nums, int target) {
        for (int i = 0; i + 1 < (int)nums.size(); i++) {
            for (int j = i + 1; j < (int)nums.size(); j++) {
                if (nums[i] + nums[j] == target) {
                    return {i, j};   // 直接构造,不用先建空 vector
                }
            }
        }
        return {};
    }
};
```

关键修改:

1. 不再先建空 vector 再下标赋值,**直接 `return {i, j}` 用列表初始化**——比 `resize` + 赋值更干净。
2. 循环条件写成 `i + 1 < nums.size()`,避开无符号减法。

## 标准做法:哈希表 O(n)

### 思路转换

暴力解法的痛点:对每个 `nums[i]`,都得"再扫一遍后面"找 `target - nums[i]`,这个"找"是 O(n)。

换个角度:**遍历到 `nums[i]` 时,我真正想问的是——"前面我见过的数里,有没有 `target - nums[i]`?"**

这是"查询是否存在 + 拿到下标"的问题,用**哈希表**就是 O(1)。

### 用 `unordered_map<int, int>`

C++ 的 `std::unordered_map` 就是哈希表。我们存:

```
key   = 数值     (nums 里的某个值)
value = 下标     (这个值出现在 nums 的哪一位)
```

为什么是 `值 → 下标`?因为我们要**用值去查**(查 `target - nums[i]`),所以"值"必须是 key。

### 设计原则:key/value 怎么选

> **把"你要用来查询的东西"放在 key 的位置。**

哈希表只对 **key** 提供 O(1) 查询。如果反过来存 `下标 → 值`,要按值查就只能遍历整张表 O(n) —— 算法直接退化回 O(n²),哈希表白用了。

写之前自问两个问题:

1. **我每次查的时候,手里有什么?** → 那个就是 key
2. **查到之后我想拿什么?** → 那个就是 value

代入两数之和:

- 手里有:`target - nums[i]`(一个**值**)
- 想拿到:那个数的**下标**

所以 `unordered_map<值, 下标>`,不能反着存。

### 算法流程

遍历数组,对每个 `nums[i]`:

1. 算"我需要的另一半":`need = target - nums[i]`
2. 查 `need` 在不在表里
   - 在 → 返回 `{表里 need 对应的下标, i}`
   - 不在 → 把 `nums[i] → i` 存进表,继续

走一遍 `nums = [2, 7, 11, 15], target = 9`:

```
i=0, nums[i]=2:  need=7,  表={}              没查到 → 表={2:0}
i=1, nums[i]=7:  need=2,  表={2:0}           查到 2! → 返回 {0, 1}
```

### 代码

```cpp
#include <unordered_map>
class Solution {
public:
    vector<int> twoSum(vector<int>& nums, int target) {
        unordered_map<int, int> seen;       // 值 -> 下标
        for (int i = 0; i < (int)nums.size(); i++) {
            int need = target - nums[i];
            auto it = seen.find(need);
            if (it != seen.end()) {
                return {it->second, i};     // it->second 是表里存的下标
            }
            seen[nums[i]] = i;              // 没找到才存自己
        }
        return {};
    }
};
```

### 三个容易踩的坑

**① 必须"先查再存",不能反过来**

考虑 `nums = [3, 3], target = 6`:

正确顺序(先查再存):
```
i=0: 查 need=3 → 表空,没查到 → 存 {3:0}
i=1: 查 need=3 → 查到 0 → 返回 {0, 1} ✅
```

错误顺序(先存再查):
```
i=0: 存 {3:0} → 查 need=3 → 查到 0 → 返回 {0, 0} ❌ 同一个元素用了两次
```

**② `find` 的返回值是迭代器**

```cpp
auto it = seen.find(need);
if (it != seen.end()) {   // end() 表示"没找到"
    int idx = it->second; // it 指向 pair<int,int>,->second 是 value
}
```

新手常写成 `if (seen[need])` —— **不要这么写**:

- `seen[need]` 如果 key 不存在,会**自动插入** `{need, 0}`,把表搞脏
- value 是 0 时 `if` 判断为 false,逻辑也错

**查存在性用 `find`,不要用 `[]`**。

**③ `count` 也行,但效率略低**

```cpp
if (seen.count(need)) {
    return {seen[need], i};
}
```

可读性更好,但查了两次哈希(`count` + `[]`);`find` + 迭代器只查一次。数据小没差别,知道就行。

### 复杂度对比

| 解法 | 时间 | 空间 |
|---|---|---|
| 暴力双循环 | O(n²) | O(1) |
| 哈希表 | **O(n)** 平均 | O(n) |

哈希表是**用空间换时间**——多用了 O(n) 内存,把每次查询从 O(n) 降到 O(1)。

---

## 收获清单

- [ ] `reserve` 不创建元素,只有 `resize` / `push_back` 会。要下标访问必须先让元素存在。
- [ ] `size_t` 是无符号的,做减法警惕下溢。
- [ ] 返回 `vector<int>` 可以直接 `return {a, b};`,不必先建容器再赋值。
- [ ] "在已遍历部分查配对值"是高频套路 → **哈希表把 O(n²) 降到 O(n)**。
- [ ] `unordered_map` 查存在性用 `find` + 迭代器,不要用 `[]`(会自动插入)。
- [ ] 哈希表"先查再存"才能避免元素自己配自己。

## 怎么才能想到用哈希表 —— 思维方法

不是"灵光一闪",是有套路的。

### ① 看穿暴力解法的"浪费"

任何 O(n²) 双循环,先问一句:**内层循环在重复做什么?**

本题内层循环在干嘛?反复扫数组找一个值:

```
i=0: 在 [7,11,15] 里找 7
i=1: 在 [11,15]   里找 2
i=2: 在 [15]      里找 -2
```

每次都是"在一堆数里找某个值"——这个动作做了 n 次,每次都从头扫。

**信号**:重复的"查找"操作 = 可能可以用某种数据结构加速。

### ② 把"我在做什么"翻译成"原语"

把内层循环用一句话概括:

> "判断某个数 `x` 是否在前面出现过,出现过的话拿到它的下标。"

拆成两个原语:

1. **存在性查询**:`x` 在不在?
2. **关联查询**:在的话,对应什么(下标)?

能 O(1) 同时回答这两个问题的数据结构,几乎只有**哈希表**。

| 你需要的能力 | 数据结构 |
|---|---|
| 只问"在不在" | `unordered_set` |
| "在不在" + "对应什么" | `unordered_map` |
| 顺序 / 有序遍历 | `set` / `map`(O(log n)) |
| 取最大/最小 | `priority_queue` |

### ③ 高频信号词

刷题时见到这些信号,立刻往哈希表上想:

| 题目里出现的信号 | 大概率用哈希表 |
|---|---|
| "找两个数,使得 ..." | 存已见过的,查互补值 |
| "判断是否出现过 / 重复" | `unordered_set` |
| "统计每个 X 出现几次" | `unordered_map<X, int>` 计数 |
| "分组 / 归类"(字母异位词) | `unordered_map<key, vector<...>>` |
| "前缀和 == 某值出现过吗" | 前缀和 + 哈希表(子数组和系列) |

两数之和是第一类的**模板题**。后面"三数之和"、"四数之和"、"和为 K 的子数组"全是它的变种。

### ④ 写完暴力后,强迫自己问三个问题

1. **内层循环在重复算什么?** 能不能预先存起来?
2. **我反复扫的是同一个集合吗?** 能不能换种表示?
3. **我用了什么信息、扔了什么?** 上一轮的信息能不能保留下来?

本题正好踩第 3 条:暴力解 `i=0` 那轮"知道了 `nums[0]` 的值"但下一轮全丢了。哈希表就是把它**留下来**。

### ⑤ 怎么练成这种直觉

没有捷径,就是量。但有几个做法让"量"更高效:

- **每题做完写一句"它属于什么套路"** —— 这份"收获清单"就是干这个的。
- **强迫自己用两种方法做** —— 做完暴力再做最优,大脑自动对比"省了什么"。
- **题目分类刷** —— 一周专攻哈希表,一周双指针,比随机刷更容易形成模式识别。
- **遇到新题先停 30 秒问**:"这题让我想起了哪道做过的?" 90% 的题都是变种。

刷到 30~50 题,看一眼题目就大概知道用什么——这不是天赋,是**模式匹配**积累出来的。
